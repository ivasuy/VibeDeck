# Dashboard — Entire Page Audit
Date: 2026-05-15

## Scope
- Page: `dashboard/src/pages/EntirePage.jsx` and checkpoint UI components under `dashboard/src/components/entire/*`.
- Focus: current UX and correctness risks after recent manual review.

## Existing Audit Summary

1. High: Race condition while loading repo data
   - In `loadRepo`, `getEntireStatus` and `getCheckpoints` run in parallel and write to shared state without a request guard.
   - If a user quickly switches repos, older responses can overwrite newer selections.
   - Location: `dashboard/src/pages/EntirePage.jsx` around `Promise.allSettled` in `loadRepo`.

2. Medium: Misleading error message on repo hide failure
   - Remove/hide failures currently fall back to `entire.repo.validation.absolute_path`, which is unrelated to hide action failures and can confuse users.
   - Location: `dashboard/src/pages/EntirePage.jsx` in `removeRepo` catch block.

3. Low: Clipboard copy failures are silent
   - `navigator.clipboard.writeText(...)` is not wrapped in async error handling; denied permission produces unhandled rejection with no user feedback.
   - Location: `dashboard/src/components/entire/CheckpointFileInspector.jsx`.

## New UX Audit Item: Checkpoint Detail Over-Exposure

Current dashboard currently renders too much raw checkpoint payload, especially for non-metadata files. This reduces signal and increases cognitive load.

- `metadata.json`:
  - Should be treated as the primary checkpoint summary source.
  - Grouped per checkpoint and shown as a single concise card containing only high-signal fields (for example: branch, provider/model, token usage, cost estimate/quality, status, session id/version if present).
- Prompt files (`prompt.txt`) :
  - Should render in a readable, formatted prompt block by default.
  - Avoid plain/raw JSON-like view where possible.
- `full.jsonl` and `content_hash.txt`:
  - Should be treated as secondary/advanced details, not primary dashboard surfaces.
  - Hide by default behind explicit expansion or separate “Advanced” section.

## Recommended UI Direction (Dashboard-safe, actionable)

- Replace “file-by-file dump” behavior with a checkpoint-card model:
  - Each checkpoint group card = one checkpoint.
  - Show group label + timestamp + usage summary + branch/model/status badges.
  - Provide nested detail sections for prompt and raw payloads.
- Keep `getCheckpoint` fetch behavior only for explicit file selection; prioritize summary fields directly from `metadata.json`.
- Provide deterministic, limited fields in preview mode:
  - status/quality, branch, provider(s), model(s), total cost, cost quality, tokens, created/updated timestamps.
- Add clear “advanced details” affordance for power users:
  - JSONL preview with pagination/line count summary only.
  - Hash file display on demand.
- Preserve existing actions (rewind/clean/enable/status) and data sources; only reshape presentation layer.

## Action Items

1. Add a request-version guard in `loadRepo` to prevent stale status/checkpoint overwrites.
2. Replace hide-repo fallback copy key with an operation-specific user message.
3. Wrap clipboard copy in try/catch and surface copy failures.
4. Implement checkpoint summary-first card layout and collapse raw file panes by default.

## Additional Second-Pass Findings

1. High: Placeholder confirm token accepted for destructive checkpoint actions
   - `/_checkConfirmToken` logs and accepts any non-empty token instead of validating it against issued confirm tokens; this defeats the destroy-action safety gate.
   - Location: `src/lib/entire-bridge.js:318-326` and usage in `/functions/vibedeck-entire/rewind` + `/functions/vibedeck-entire/clean` in `src/lib/local-api.js:2352-2442`.

2. High: Checkpoint usage hydration is synchronous and potentially expensive at scale
   - `/functions/vibedeck-checkpoints` builds per-group usage by iterating and awaiting each checkpoint metadata/db read in sequence, so repo-scale checkpoints can block response time and increase memory/CPU usage.
   - Location: `src/lib/local-api.js:2260-2283` and `src/lib/entire-checkpoint-usage.js:501-533`.

3. Medium: Path validation is narrowly slash-based and may diverge from cross-platform expectations
   - Both `isValidCheckpointPath` and backend `readCheckpoint` validate traversal only via `/` segments, which is brittle for platform/path encoding edge cases and can behave inconsistently for paths with backslashes/encoded separators.
   - Location: `src/lib/local-api.js:1750-1757` and `src/lib/entire-bridge.js:91-97`.

4. Medium: Cached repo-state fetch remains alias-fragile
   - Cached status path is populated from `repo_root` aliases on write path, but lookup still relies on a single exact match, so a valid repo can miss cache when canonical/raw representations differ.
   - Location: `src/lib/local-api.js:2555-2561` and `src/lib/db/repos.js:36-40` plus alias expansion in `src/lib/db/repos.js:6-16`.

5. Medium: Configure command parser can mis-handle quoting
   - `parseArgv` in the Configure panel splits by whitespace only, so quoted shell-like args are broken into multiple entries and can call `entire configure` with unintended arguments.
   - Location: `dashboard/src/components/entire/AdvancedConfigurePanel.jsx:8-12`.

6. Medium: Large raw checkpoint payloads render unbounded in the inspector
   - The preview and raw tabs render full file content (and JSON.stringify of parsed payloads) without length caps, so large metadata/content files can degrade UI performance.
   - Location: `dashboard/src/components/entire/CheckpointFileInspector.jsx:142-167,175-177`.

7. Medium: Parse failure state is not surfaced in inspector
   - JSON parse failures are partially recorded in payload (`parse_error`) but not shown in the UI, leaving users with silently incomplete object previews and little diagnostic context.
   - Location: `src/lib/entire-bridge.js:169-175` and `dashboard/src/components/entire/CheckpointFileInspector.jsx:15-18,147-149`.

8. Low: Clipboard failure remains silent in UI copy action
   - Copy action calls `navigator.clipboard.writeText` with no async error handling or status feedback.
   - Location: `dashboard/src/components/entire/CheckpointFileInspector.jsx:134`.
