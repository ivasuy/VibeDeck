# Dashboard Entire Follow-Up Audit
Date: 2026-05-16

## Scope
- Branch: `entire/ui-fix`
- Focus: remaining work after the `/entire` dashboard UI revamp.
- Constraint: audit only; no implementation changes in this pass.

## Status After UI Revamp

The main `/entire` dashboard clutter issue is largely addressed in the current branch.

- `dashboard/src/pages/EntirePage.jsx` now uses `EntireCommandCenter`, `CheckpointTimeline`, and `EntireControlPanel` instead of the old file-first checkpoint list.
- `loadRepo` now has a request sequence guard, so stale status/checkpoint responses should not overwrite a newer selected repo.
- `CheckpointTimeline` and `CheckpointCard` now present checkpoint groups as cards with accumulated metadata, usage, cost, prompts, and captured activity.
- Prompt content is collapsed by default.
- Metadata and file lists are moved under advanced details instead of being the primary UI surface.
- JSONL activity is summarized into event counts instead of being shown as a raw file dump in the primary flow.

Remaining work should avoid reworking the new dashboard layout unless a specific regression appears. The highest-value next pass is targeted correctness and polish around backend safety, cache/path handling, and the older inspector/configure surfaces.

## Remaining Findings

### 1. High: Bridge-level destructive confirm token remains a trust boundary gap

`src/lib/local-api.js` validates and consumes issued confirm tokens before calling rewind/clean, which protects the normal HTTP path. However, `src/lib/entire-bridge.js` still accepts any non-empty confirm token in `_checkConfirmToken`.

Current risk:
- Direct or future internal callers of `rewindCheckpoint` / `cleanEntire` can bypass the real confirm-token store by passing any non-empty string.
- The bridge logs a placeholder warning, which means the lower-level API contract still does not enforce the same safety invariant as the HTTP route.

Recommended fix:
- Move real confirm-token validation into the boundary that executes destructive operations, or split the bridge API so only already-authorized call sites can reach destructive helpers.
- Add tests for missing token, invalid token, expired token, wrong operation token, valid single-use token, and double-use rejection.

Breakage risk:
- Medium. A strict bridge-level validation can accidentally block valid rewind/clean flows if the HTTP layer consumes the token before the bridge validates it. The fix needs one clear ownership point for token consumption, not duplicate consumption in both layers.

### 2. High: Checkpoint usage hydration can be expensive at scale

`/functions/vibedeck-checkpoints` builds checkpoint usage synchronously before returning the response. `buildCheckpointUsageIndex` iterates each checkpoint group and awaits metadata reads one group at a time.

Current risk:
- Large repos with many Entire checkpoints can make `/entire` slow to load.
- The endpoint couples list rendering to usage hydration, so a heavy usage scan can block even basic checkpoint navigation.
- Sequential reads leave no timeout, cap, concurrency limit, or partial-response behavior.

Recommended fix:
- Keep the response shape backward-compatible.
- Add bounded concurrency and a maximum hydration budget.
- Consider returning checkpoint files first with usage marked as partial/loading, then hydrate usage through a follow-up endpoint or cached background pass.
- Cache usage summaries by checkpoint id plus metadata/content hash when possible.

Breakage risk:
- Medium-high. This is the most likely remaining item to change visible data. A bad implementation could show missing costs/tokens or make cards flicker. It needs fixture tests for small, large, partial, and malformed checkpoint groups.

### 3. Medium: Checkpoint path validation is too slash-specific

`isValidCheckpointPath` rejects absolute paths and `..` segments only after splitting on `/`. The bridge-side read path has the same general shape.

Current risk:
- Backslash paths, encoded traversal, mixed separators, and platform-specific path behavior may not be handled consistently.
- This is both a safety and compatibility issue because the dashboard reads checkpoint files by user-visible relative paths.

Recommended fix:
- Normalize separators before validation.
- Reject encoded traversal and NUL bytes.
- Resolve the requested checkpoint path against the checkpoint root and verify it remains inside that root.
- Add tests for normal relative paths, nested checkpoint files, `../`, `..\\`, encoded traversal, absolute Unix paths, absolute Windows-style paths, and NUL.

Breakage risk:
- Medium. Too strict a validator can block legitimate checkpoint files; too loose a validator keeps traversal risk. The tests should define the intended allowed path format before code changes.

### 4. Medium: Cached repo-state lookup remains alias-fragile

Repo state writes keep aliases, but cached status lookup still calls `getRepoState(dbPath, repoRoot)` with one exact repo string.

Current risk:
- The same repo can miss cached status when accessed through raw path, realpath, symlink path, or another alias.
- Users may see stale or empty cached Entire state even though the repo was previously enabled or checked.

Recommended fix:
- Make `getRepoState` alias-aware using the same alias expansion used on write.
- Prefer canonical realpath when available, but preserve raw-path compatibility.
- Add tests for raw path vs realpath/symlink alias reads.

Breakage risk:
- Low-medium. Main risk is returning the wrong row if aliases are too broad. Keep aliases limited to raw path and `fs.realpathSync` output.

### 5. Medium: Configure command parser does not support shell-style quoting

`AdvancedConfigurePanel` uses `text.match(/\S+/g)`, so quoted values are split into separate args.

Current risk:
- Users can unintentionally send malformed `entire configure` arguments when values include spaces.
- This is a UI correctness issue, not a backend issue.

Recommended fix:
- Add a small argv parser that supports quotes and escaped quotes, or require one-argument-per-line input.
- Surface parse errors before making the command call.
- Add tests for simple args, quoted values, escaped quotes, empty input, and unmatched quotes.

Breakage risk:
- Low-medium. Existing simple whitespace args must behave exactly as before. Quoted parsing should only expand valid use cases.

### 6. Medium: Old checkpoint file inspector still renders raw payloads without caps

The new `/entire` primary flow reduces raw data exposure, but `CheckpointFileInspector` still renders raw content and full parsed JSON when used.

Current risk:
- If this inspector is still reachable or reused later, very large `metadata.json`, `full.jsonl`, or raw files can degrade browser performance.
- Parse failures are recorded in payloads but are not clearly surfaced in this inspector.

Recommended fix:
- Add preview caps for raw and parsed views.
- Show file size/line count and a clear "truncated" state.
- Surface `parse_error` when present.
- Keep full content behind an explicit copy/download-style action only if needed.

Breakage risk:
- Low. This can be implemented as defensive UI behavior without changing backend data contracts.

### 7. Low: Hide-repo fallback error message is still misleading

`removeRepo` still falls back to `entire.repo.validation.absolute_path` when hide/remove fails for a non-Error reason.

Current risk:
- Users can see an absolute-path validation message for an operation that is actually a hide/remove failure.

Recommended fix:
- Add an operation-specific copy key such as `entire.repo.remove.error_fallback`.
- Use that key only in the hide/remove catch path.

Breakage risk:
- Low. Copy-only change plus a small assertion is enough.

### 8. Low: Clipboard copy failure remains silent in the old inspector

`CheckpointFileInspector` still calls `navigator.clipboard?.writeText(file.raw || "")` without async error handling.

Current risk:
- Browser permission denial or unavailable clipboard silently fails.

Recommended fix:
- Wrap copy in async try/catch.
- Show a short success/failure status in the card.

Breakage risk:
- Low. This is isolated UI state.

## Recommended Order

1. Copy/error polish: hide-repo fallback, clipboard status, parse-error display, raw preview caps.
2. Configure parser with tests.
3. Alias-aware repo-state lookup.
4. Path validation hardening.
5. Destructive confirm-token contract cleanup.
6. Checkpoint usage hydration performance.

This order keeps low-risk UI correctness first, then moves into cache/path correctness, and leaves the larger backend contract/performance changes for focused test-driven passes.
